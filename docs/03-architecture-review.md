# Wishify — Архитектурный разбор

> Статус: черновик v0.1 | Дата: 2026-03-14

---

## 1. Спорные решения

### 1.1 Хранение токенов — localStorage vs httpOnly cookie

**Ситуация:** В `08-architecture.md` описаны два варианта без выбора:
`access_token в localStorage + refresh_token в httpOnly cookie` или `оба в localStorage`.

**Проблема:** `localStorage` уязвим к XSS. Если любой скрипт на странице получит доступ к токенам — авторизация скомпрометирована. Для вишлиста с публичными ссылками, скрейпером URL и guest-именами риск выше, чем обычно.

**Рекомендация:** `refresh_token` — httpOnly cookie, `access_token` — только в памяти (React Context). Да, сложнее с CORS, но это единственный безопасный вариант.

---

### 1.2 Soft-delete: только при взносах или при любой активности?

**Ситуация:** Правило гласит: «товар с взносами нельзя удалить жёстко». Но если у товара есть активная **резервация** без взносов (is_group_fund = false) — правило молчит.

**Проблема:** Hard-delete товара с активной резервацией уничтожит резервацию через CASCADE DELETE. Гость, резервировавший подарок, ничего не узнает — у него нет email. WebSocket разошлёт `item.deleted`, но этого недостаточно — данные пропадут.

**Рекомендация:** Условие soft-delete: `deleted_at IS NOT NULL` если есть хотя бы одна запись в `reservations WHERE released_at IS NULL` **или** в `contributions` для этого `item_id`. Предупреждение в UI показывать в обоих случаях.

---

### 1.3 Гостевой резерв при потере localStorage — «мёртвая блокировка»

**Ситуация:** Гость резервирует товар, закрывает браузер, токен потерян. `guest_token` нет — снять резерв нельзя. Резервация в БД остаётся активной (`released_at IS NULL`). Товар заблокирован навсегда.

**Проблема:** Это не просто «потеря резерва» — это постоянная блокировка подарка для всех остальных, даже для зарегистрированных пользователей.

**Варианты решения:**
- Владелец списка может принудительно снять резерв с любого товара (эндпоинт с проверкой owner).
- TTL на гостевые резервации (например, 30 дней без активности `guest_session.last_seen_at`).

**Рекомендация:** Добавить `DELETE /api/v1/lists/{list_id}/items/{item_id}/reserve` для владельца без проверки «ты ли резервировал» + автоматическое освобождение через cron/фоновую задачу.

---

### 1.4 Взносы сверх цели — нет ограничений

**Ситуация:** «Показываем цель достигнута, не блокируем». Но контракт не определяет: можно ли вносить бесконечно после `total_contributed >= target_amount`?

**Проблема:** Пользователи могут внести в 10 раз больше цели. Это запутывает и может создать неловкость (кто виноват, что перебрали?).

**Рекомендация:** Явно зафиксировать решение в контракте. Вариант — мягкое предупреждение при внесении взноса, если после него `total_contributed > target_amount`, но не блокировать.

---

### 1.5 Определение caller_role — нет явного контракта

**Ситуация:** `GET /api/v1/lists/public/{slug}` возвращает `caller_role: 'owner' | 'viewer'`, от которого зависит фильтрация данных. Но алгоритм определения роли нигде не описан явно.

**Проблема:** Граничные случаи не покрыты:
- Авторизованный пользователь, но не владелец → viewer?
- Гость открывает публичную ссылку → anon или viewer?
- Владелец с истёкшим JWT → viewer или 401?

**Уточнение (см. раздел 2).**

---

### 1.6 contributions.CASCADE DELETE при hard-delete

**Ситуация:** В схеме: `contributions.item_id FK → items.id CASCADE DELETE`. Жёсткое удаление товара уничтожит взносы. Логика в коде проверяет наличие взносов перед удалением, но:

**Проблема:** Между проверкой и DELETE возможен race condition — новый взнос успевает попасть в БД. В итоге взнос удаляется без предупреждения пользователя.

**Рекомендация:** Проверку и DELETE выполнять в одной транзакции с `SELECT ... FOR UPDATE` на `item_id`. Или перенести CASCADE на уровень сервиса (без ON DELETE CASCADE в БД), делая явный DELETE contributions перед item.

## 2. Уточнённые контракты и границы

### 2.1 Определение caller_role

```
GET /api/v1/lists/public/{slug}
```

| Контекст запроса | caller_role |
|---|---|
| Нет токена | `viewer` |
| `X-Guest-Token` | `viewer` |
| `Authorization: Bearer` + `user.id == list.owner_id` | `owner` |
| `Authorization: Bearer` + `user.id != list.owner_id` | `viewer` |
| Истёкший JWT | `viewer` (не 401 — публичный эндпоинт) |

**Правило:** Публичный эндпоинт никогда не возвращает 401. Невалидный токен трактуется как «нет токена».

---

### 2.2 Двухфазный DELETE /items/{item_id}

```
Фаза 1 — проверка:
  DELETE /items/{item_id}            (без confirm)
  → если есть взносы ИЛИ резерв:    200 { warning: "has_contributions" | "has_reservation" | "has_both" }
  → иначе:                           hard-delete → 204

Фаза 2 — подтверждение:
  DELETE /items/{item_id}?confirm=true
  → soft-delete (deleted_at = NOW()) → 200 { deleted: true, soft: true }
```

**Инвариант:** `?confirm=true` всегда делает soft-delete, никогда hard-delete, если была активность.

**Граница:** Между фазой 1 и 2 состояние могло измениться. Бэкенд на фазе 2 снова проверяет условие и делает soft-delete независимо от «предыстории».

---

### 2.3 WebSocket — разграничение ролей

Redis `payload_full` содержит всё. Каждый воркер при отправке каждому клиенту обязан пропустить payload через фильтр.

```
Поле в WS-событии           | owner | viewer |
----------------------------|-------|--------|
item_id                     |  ✅   |   ✅   |
is_reserved                 |  ✅   |   ✅   |
reserved_by_me              |  ❌   |   ✅   |
reserver_name               |  ❌   |   ✅   |
total_contributed           |  ✅   |   ✅   |
contribution_count          |  ✅   |   ✅   |
contributor_name            |  ❌   |   ❌   | ← НИКТО не видит
my_contributions            |  ❌   |   ✅ (только свои) |
```

**Граница:** `contributor_name` не отдаётся никому через WS — только через `/contributions/summary` с фильтром `my_contributions`.

---

### 2.4 Гостевая идентификация — граница ответственности

| Уровень | Обязанность |
|---|---|
| `localStorage` (клиент) | Хранить `guest_token` |
| `GuestContext` (React) | Читать токен, показывать GuestBanner |
| `X-Guest-Token` header | Передавать токен в каждом запросе |
| `guest_sessions` (БД) | Источник истины — кто есть кто |
| Backend `deps.py` | Валидировать токен, возвращать `GuestSession` или 401 |

**Правило:** Фронтенд не принимает решений об идентификации — только передаёт токен. Бэкенд не хранит токен нигде, кроме `guest_sessions.token`.

---

### 2.5 Порядок операций при любом изменении состояния

```
1. Валидация входных данных (Pydantic)
2. Проверка прав доступа (deps.py)
3. Бизнес-логика в service (включая проверку конфликтов)
4. DB commit (AsyncSession)
5. Redis PUBLISH (после commit, не до)
6. Вернуть HTTP-ответ клиенту
```

**Инвариант:** Redis PUBLISH всегда после DB commit. Никогда до. При ошибке commit — publish не происходит.

---

### 2.6 Scraper — границы безопасности

```
POST /api/v1/util/scrape { url }
```

- Rate limit: 10 req/min per user (зарегистрированный) / 3 req/min per IP (гость — если разрешено)
- SSRF-защита: запрещены IP-адреса из RFC 1918 (10.x, 172.16-31.x, 192.168.x), localhost, `.internal`
- Таймаут httpx: 10 секунд
- Кэш: `url_scrape_cache` — TTL 24 часа (поле `scraped_at`, проверять при чтении)
- При 403/timeout/любой ошибке: `{ title: null, image_url: null, price: null }` + 200 (не ошибка)

## 3. Архитектурные инварианты

> Нарушение любого из этих правил — баг уровня blocker.

---

### INV-01 — Конфиденциальность: фильтрация только на бэкенде

Владелец списка **никогда** не получает:
- имена и идентификаторы резервирующих
- имена и суммы вкладчиков

Проверка происходит **исключительно на бэкенде** — в `list_service.py` и `ws_manager.py`. Фронтенд не вправе полагаться на то, что «сам не покажет». Данные просто не приходят в ответе.

```
Тест: owner делает GET /api/v1/lists/public/{slug}
      → в ответе нет полей reserver_name, reserved_by, contributor_name, my_contributions
```

---

### INV-02 — Атомарное резервирование через UNIQUE INDEX

```sql
UNIQUE INDEX (item_id) WHERE released_at IS NULL
```

Этот индекс — единственная гарантия от двойного резервирования. Его нельзя удалять, обходить через `ON CONFLICT DO NOTHING` без возврата 409, или заменять проверкой в коде.

```
При конфликте INSERT → PostgreSQL UniqueViolation → 409 Conflict
                                                   + WS event "error" с пояснением
```

---

### INV-03 — Soft-delete сохраняет историю

Товар с взносами **или** активной резервацией не удаляется физически. `deleted_at` ставится — строка остаётся. Взносы и резервации в БД сохраняются. Это гарантирует целостность финансовой истории.

```
Нельзя: DELETE FROM items WHERE id = ?  -- если есть contributions или reservations
Можно:  UPDATE items SET deleted_at = NOW() WHERE id = ?
```

---

### INV-04 — Redis PUBLISH только после DB commit

```python
await session.commit()       # сначала коммит
await redis.publish(...)     # потом публикация
```

Если publish произойдёт до commit — клиенты получат событие о данных, которых ещё нет в БД. При немедленном refetch получат старое состояние. Это приводит к UI-рассинхрону, который очень трудно отловить.

---

### INV-05 — Роль определяется на бэкенде, не на фронтенде

Фронтенд передаёт токен. Бэкенд определяет `caller_role` и формирует ответ. Фронтенд **не вправе** решать «я сейчас owner, поэтому спрячу это поле» — он просто рендерит то, что получил.

---

### INV-06 — WS-фильтр применяется per-connection

Каждый WebSocket-клиент получает payload, собранный **индивидуально** для его роли и identity. Нельзя отправить один payload broadcast всем — сначала фильтр, потом отправка каждому.

---

### INV-07 — Публичный эндпоинт не возвращает 401

`GET /api/v1/lists/public/{slug}` доступен без авторизации. Невалидный или истёкший токен → трактуется как «нет токена» → `caller_role = viewer`. Ошибка аутентификации не блокирует просмотр.

---

### INV-08 — Секреты только через переменные окружения

JWT_SECRET, DATABASE_URL, REDIS_URL, GOOGLE_CLIENT_SECRET — никогда в коде, никогда в git. Нарушение = публичный деплой с известным секретом.

---

### INV-09 — Все денежные суммы хранятся как NUMERIC(12,2)

Нельзя использовать `float` или `int` для хранения и расчёта денежных значений. `NUMERIC` в PostgreSQL, `Decimal` в Python. Конвертация в `float` только для отображения, никогда для записи.

---

### INV-10 — guest_token не передаётся по незащищённым каналам

`X-Guest-Token` принимается только по HTTPS (в production). Локально (HTTP) допустимо. Бэкенд не логирует токены.

## 4. Открытые вопросы (требуют решения до реализации)

| # | Вопрос | Влияние |
|---|---|---|
| Q1 | Хранить refresh_token в httpOnly cookie или localStorage? | Безопасность, сложность CORS |
| Q2 | Может ли владелец принудительно снять чужой резерв? | INV на «мёртвую блокировку» гостя |
