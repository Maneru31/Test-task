## 2. Контекст проекта

**Продукт:** Пользователь создаёт список желаний, добавляет товары (название, ссылка, цена, фото) и делится по ссылке. Друзья резервируют подарки и скидываются на дорогие. Владелец **никогда** не видит кто что зарезервировал и кто сколько внёс.

**Ключевые требования:**
- Публичная ссылка работает без регистрации
- Реалтайм обязателен (без перезагрузки страницы)
- Мобильный адаптив (320px+)
- Деплой — работающее приложение, не локальный прототип

## 4. Стандарты качества

### 4.1 Общие
- Нет заглушек и `# TODO` — если не реализовано, это блокер, а не TODO
  - Исключение: явно оговорённые заглушки с пометкой `# STUB: причина`
- Нет дублирования кода — выноси в утилиты/сервисы
- Нет скрытых допущений — если не уверен, перечисли варианты явно
- Секреты только через env-переменные, никогда в коде или комментариях

### 4.2 Backend (Python)
- Все endpoint-функции — `async def`
- Все DB-операции через `AsyncSession`, никакого синхронного SQLAlchemy
- Pydantic-схемы для всех request/response — никаких голых `dict`
- Бизнес-логика в `app/services/`, не в роутерах
- Зависимости FastAPI — в `app/core/deps.py`
- `logging` вместо `print()`
- Правильные HTTP-статусы: 201 (создание), 204 (удаление без тела), 409 (конфликт), 422 (валидация)
- `HTTPException` с понятным `detail` на русском (пользователь его увидит) или коде (для клиента)

### 4.3 Frontend (TypeScript/Next.js)
- Server components (`async page`) — там где данные нужны при загрузке
- `"use client"` — только при нужде в хуках или интерактивности
- TanStack Query — для всех запросов к API. Никакого `useEffect` + `useState` для фетчинга
- `api.ts` — единственное место с `baseURL` и interceptors
- Типы — в `src/types/index.ts`, только named exports
- Компоненты: PascalCase файлы (`WishItem.tsx`)
- Хуки: camelCase с `use` (`useListWebSocket.ts`)
- Никакого `any` без явного обоснования в комментарии

### 4.4 Тесты
Писать тесты там, где это критично для поведения/контрактов:

| Где | Что покрывать тестами |
|---|---|
| `reservations_service` | race condition, 409 при конфликте, release |
| `list_service` | фильтрация ответа по роли (owner vs viewer) |
| `auth_service` | регистрация, логин, невалидный токен |
| `scraper_service` | SSRF защита, fallback при 403 |
| WS `ws_manager` | payload фильтрация owner не видит имён |

Для бэкенда: `pytest` + `pytest-asyncio` + `httpx.AsyncClient`.
Не мокать БД там, где проверяется поведение уникальных индексов.

## 5. Бизнес-правила (нарушать нельзя)

### 5.1 Конфиденциальность (КРИТИЧНО)
Владелец списка **никогда** не видит:
- Кто зарезервировал подарок (ни имя, ни `user_id`)
- Кто и сколько внёс в групповой сбор

Фильтрация — на **бэкенде** по роли (`owner` / `viewer`). Не на фронтенде.
Проверяется в каждом endpoint И в каждом WebSocket-событии.

```python
# Правило для каждого endpoint/WS event с reservation или contribution:
if caller.is_owner:
    # только агрегаты: is_reserved, total_contributed, contribution_count
else:
    # + reserved_by_me, reserver_name, my_contributions
```

### 5.2 Реалтайм (ОБЯЗАТЕЛЬНО)
Любое действие (резерв, взнос, добавление/удаление товара) мгновенно видно всем.

Цепочка: `REST commit → Redis PUBLISH → WS broadcast (filtered per viewer)`

Клиент патчит TanStack Query cache через `queryClient.setQueryData` — без перезапроса к API.

### 5.3 Гостевой доступ
- Публичная ссылка: без регистрации
- Перед первым действием → модал «Как вас зовут?»
- `guest_token` → `localStorage`
- Все запросы гостя с `X-Guest-Token` header
- UI предупреждает: «без аккаунта сессия не сохраняется»

### 5.4 Race condition при резервировании
`UNIQUE INDEX (item_id) WHERE released_at IS NULL` на уровне БД — atomic, без locks.
Конфликт → 409 + WS `error` event всем, кто смотрел на этот товар.

### 5.5 Soft-delete
Товар с взносами → только `deleted_at`, никаких CASCADE DELETE.
Перед удалением с взносами → предупреждение в UI с количеством вкладчиков.
Soft-deleted товары не попадают в API-ответы и WS.

## 7. Edge cases — принятые решения

| Ситуация | Решение |
|---|---|
| Товар удалён, пока на него скидывались | Soft-delete. Предупреждение при удалении: «X человек уже внесли вклад» |
| Два человека резервируют одновременно | DB unique index → первый побеждает, второй получает 409 + WS уведомление |
| Взносы превысили цель | Разрешить, показать «Цель достигнута» зелёным |
| Цель не набрана к дате | Информационно, не блокировать |
| Дата события в прошлом | Баннер «Событие прошло», список остаётся доступным |
| Владелец открывает публичную ссылку | Backend убирает все имена из ответа (детектирует по JWT `user_id == owner_id`) |
| Гость потерял localStorage | Потерял доступ к своим резервам. Нет recovery — по дизайну |
| URL scraper → 403 / timeout | Вернуть частичные данные, предложить заполнить вручную |
| WS разорван | Exponential backoff: 1s → 2s → 4s → ... → 30s. При реконнекте — `invalidateQueries` |
| SSRF в scraper | Блокируем `10.x`, `192.168.x`, `127.x`, только HTTPS |
