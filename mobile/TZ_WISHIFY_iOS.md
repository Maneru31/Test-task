## 1. Назначение документа

Документ описывает функциональные и нефункциональные требования к мобильному iOS-приложению **Wishify**. Приложение является нативным мобильным клиентом к существующему веб-сервису Wishify, предназначенному для создания и совместного использования вишлистов.

Документ является основой для:
- разработки (frontend mobile)
- ревью (code review, архитектурные решения)
- QA-тестирования
- приёмки готового продукта

---

## 2. Описание продукта

### 2.1. Что такое Wishify

Wishify — сервис для создания списков желаний. Пользователь создаёт список, добавляет в него товары (с ценой, ссылкой, изображением), и делится публичной ссылкой с друзьями и родственниками. Друзья могут анонимно зарезервировать позиции или скинуться на дорогой подарок. Владелец не видит, кто что зарезервировал — сюрприз сохраняется.

### 2.2. Ключевые сценарии

| # | Сценарий | Актор |
|---|---------|-------|
| S1 | Создать список и добавить желания | Авторизованный пользователь |
| S2 | Поделиться ссылкой на список | Авторизованный пользователь |
| S3 | Открыть чужой список по ссылке и зарезервировать подарок | Гость / авторизованный |
| S4 | Скинуться на групповую покупку | Гость / авторизованный |
| S5 | Видеть обновления списка в реальном времени | Все пользователи |

### 2.3. Связь с вебом

Мобильное приложение использует **тот же бэкенд** что и веб-версия. API-контракты не меняются. Изменения бэкенда в рамках данного ТЗ не предусмотрены, кроме одного опционального улучшения (п. 9.1).

## 3. Целевая аудитория

### 3.1. Основные пользователи

**Именинник / именинница** — создаёт список, делится с близкими. Ожидает простого добавления желаний (в том числе через вставку ссылки на товар) и удобного шеринга.

**Друг / родственник** — открывает чужой список по ссылке, выбирает подарок, резервирует или вносит вклад. Может не иметь аккаунта.

### 3.2. Устройства

- iPhone с iOS 16.0 и выше
- Поддержка iPhone SE (4.7") до iPhone 15 Pro Max (6.7")
- Только портретная ориентация (landscape — не поддерживается в MVP)

## 5. Функциональные требования

> Приоритет: **[M]** Must Have, **[S]** Should Have, **[C]** Could Have.

### 5.1. Аутентификация

#### FR-01 — Регистрация по email [M]

**Описание:** Пользователь может зарегистрироваться с помощью email и пароля.

**Поля формы:**
- Отображаемое имя (display_name): обязательное, 2–50 символов
- Email: обязательное, валидный формат email
- Пароль: обязательное, минимум 8 символов

**Поведение:**
- При нажатии «Создать аккаунт» → `POST /auth/register`
- При успехе → сохранить access_token в Keychain → перейти на Dashboard
- При ошибке (email уже занят) → показать inline-ошибку под полем email
- Кнопка «Войти» → переход на LoginScreen
- Все поля валидируются до отправки запроса (Zod)

**Макет:** Поля сверху вниз: имя → email → пароль → кнопка «Создать аккаунт» → ссылка «Уже есть аккаунт? Войти»

---

#### FR-02 — Вход по email [M]

**Описание:** Пользователь может войти с помощью email и пароля.

**Поля формы:**
- Email: обязательное
- Пароль: обязательное

**Поведение:**
- При нажатии «Войти» → `POST /auth/login`
- При успехе → сохранить access_token в Keychain → Dashboard
- При ошибке 401 → «Неверный email или пароль»
- Кнопка «Регистрация» → RegisterScreen
- Кнопка «Войти через Google» → FR-03

---

#### FR-03 — Google OAuth [M]

**Описание:** Пользователь может войти через аккаунт Google.

**Поведение:**
- Нажатие кнопки «Войти через Google» открывает `SFSafariViewController`
- После авторизации → Google перенаправляет на `wishify://oauth/callback?access_token=...`
- Приложение перехватывает deep link → извлекает токен → сохраняет в Keychain → Dashboard
- Если Google OAuth недоступен (нет сети) → показать ошибку

---

#### FR-04 — Выход [M]

**Описание:** Пользователь может выйти из аккаунта.

**Поведение:**
- Нажатие «Выйти» на экране Профиля → показать Alert подтверждения
- При подтверждении → `POST /auth/logout` (best-effort) → удалить токен из Keychain → перейти на LoginScreen
- Если запрос logout упал (нет сети) → всё равно разлогинить локально

---

#### FR-05 — Автологин при открытии [M]

**Описание:** При открытии приложения пользователь не должен вводить пароль повторно, если токен актуален.

**Поведение:**
- При старте → читать access_token из Keychain
- Если токен есть → `GET /auth/me` → при успехе → Dashboard
- Если `GET /auth/me` вернул 401 → попытка refresh (`POST /auth/refresh`) → при успехе → Dashboard
- Если refresh не удался → LoginScreen
- Если токена нет → LoginScreen
- Пока идёт проверка → показывать Splash screen с логотипом

---

### 5.2. Мои списки (Dashboard)

#### FR-06 — Просмотр списков [M]

**Описание:** Авторизованный пользователь видит все свои вишлисты.

**Поведение:**
- При открытии → `GET /lists` → отобразить сетку карточек (2 колонки)
- Каждая карточка содержит: название, повод (occasion), дата повода, количество позиций
- Pull-to-refresh обновляет список
- Если списков нет → EmptyState с иллюстрацией и кнопкой «Создать первый список»
- Состояние загрузки → Skeleton cards (2 колонки, 4 карточки)

---

#### FR-07 — Создание списка [M]

**Описание:** Пользователь может создать новый вишлист.

**Поведение:**
- Кнопка «+» в хедере Dashboard → открыть BottomSheet «Новый список»
- Поля:
  - Название: обязательное, 1–100 символов
  - Описание: необязательное, до 500 символов
  - Повод: необязательное, Picker (День рождения / Новый год / Свадьба / Другое)
  - Дата: необязательная, DatePicker (только будущие даты или текущая)
- Кнопка «Создать» → `POST /lists` → закрыть BottomSheet → добавить карточку в начало сетки
- Оптимистичное обновление: карточка появляется мгновенно, при ошибке — откатывается

---

#### FR-08 — Удаление списка [M]

**Описание:** Пользователь может удалить список.

**Поведение:**
- Swipe влево на карточке → кнопка «Удалить» (красная)
- Либо долгое нажатие → контекстное меню с «Удалить»
- Alert подтверждения: «Удалить список "[название]"? Это действие нельзя отменить»
- При подтверждении → `DELETE /lists/{id}` → оптимистично убрать карточку из сетки
- При ошибке → вернуть карточку + toast «Не удалось удалить»

---

#### FR-09 — Переход в список [M]

**Описание:** Нажатие на карточку открывает экран деталей списка.

**Поведение:** Tap → `ListDetailScreen` с `listId` в параметрах.

---

### 5.3. Список желаний (owner view)

#### FR-10 — Просмотр позиций [M]

**Описание:** Владелец видит все позиции своего списка.

**Поведение:**
- `GET /lists/{listId}` → показать позиции в вертикальном списке (FlashList)
- Каждая позиция-карточка содержит:
  - Изображение (FastImage, placeholder если нет)
  - Название
  - Цена и валюта (форматированные: «1 500 ₽»)
  - Признак групповой покупки (иконка)
  - Кнопки «Редактировать» и «Удалить»
  - Иконка drag-handle для перетаскивания
- Pull-to-refresh
- Skeleton-лоадер при первой загрузке

**Хедер экрана:**
- Кнопка «< Назад» (Dashboard)
- Название списка
- Кнопки «Поделиться» и «Настройки» (иконки)

**Футер:**
- Кнопка «+ Добавить желание» → `AddItemScreen`

---

#### FR-11 — Добавление позиции [M]

**Описание:** Владелец может добавить новое желание в список.

**Поля формы:**
- URL: необязательное — при вставке запускается скрапинг (FR-17)
- Изображение: превью (из URL или пустой placeholder)
- Название: обязательное, 1–200 символов
- Описание: необязательное, до 1000 символов
- Цена: необязательная, числовое, ≥ 0
- Валюта: Picker (RUB / USD / EUR), по умолчанию RUB
- Групповая покупка: Switch (вкл/выкл)
- Целевая сумма: показывается только если «Групповая покупка» = вкл, числовое, > 0

**Поведение:**
- Кнопка «Сохранить» → `POST /lists/{listId}/items` → закрыть экран → позиция появляется в конце списка
- Все поля валидируются до отправки
- При ошибке → toast с текстом ошибки, форма остаётся открытой

---

#### FR-12 — Редактирование позиции [M]

**Описание:** Владелец может изменить существующую позицию.

**Поведение:**
- Открывает `EditItemScreen` с предзаполненными данными
- `PATCH /lists/{listId}/items/{itemId}`
- При успехе → вернуться на ListDetailScreen → обновить карточку

---

#### FR-13 — Удаление позиции [M]

**Описание:** Владелец может удалить позицию из списка.

**Поведение:**
- Кнопка «Удалить» на карточке → Alert подтверждения
- `DELETE /lists/{listId}/items/{itemId}`
- Оптимистичное удаление → при ошибке откат + toast

---

#### FR-14 — Настройки списка [M]

**Описание:** Владелец может изменить метаданные списка.

**Поля:** те же, что при создании (название, описание, повод, дата).

**Поведение:**
- `PATCH /lists/{listId}`
- «Опасная зона»: кнопка «Удалить список» (красная, с подтверждением)

---

### 5.4. Управление позициями

#### FR-15 — Перетаскивание позиций [M]

**Описание:** Владелец может менять порядок позиций drag-and-drop.

**Поведение:**
- Долгое нажатие на drag-handle активирует drag
- При отпускании → оптимистично обновить порядок в UI
- `PATCH /lists/{listId}/items/reorder` с массивом item_ids (дебаунс 500мс)
- При ошибке → откатить порядок + toast

---

#### FR-16 — Автозаполнение из URL [M]

**Описание:** При вставке URL в поле ссылки — автоматически заполнить название, изображение и цену.

**Поведение:**
- onChange поля URL с дебаунсом 800мс
- Если значение является валидным URL → `POST /items/scrape { url }`
- Показать ActivityIndicator рядом с полем URL
- При ответе → автозаполнить поля (если они ещё пустые)
- Если пользователь уже заполнил поле вручную — не перезаписывать
- При ошибке скрапинга → silently fail (не показывать ошибку)

### 5.5. Публичный список (viewer/guest view)

#### FR-18 — Просмотр публичного списка [M]

**Описание:** Любой пользователь (авторизованный или гость) может открыть список по публичной ссылке.

**Поведение:**
- `GET /lists/public/{slug}` → показать список позиций
- Каждая карточка:
  - Изображение, название, описание
  - Цена
  - Статус: «Свободно» / «Зарезервировано» (с именем резервирующего)
  - Кнопка «Зарезервировать» (если свободно и пользователь не резервировал)
  - Кнопка «Отменить резервацию» (если пользователь сам резервировал)
  - Для групповой покупки: прогресс-бар + «Участвовать»
- Pull-to-refresh
- Владелец **не видит** имена резервирующих (бэкенд фильтрует)

---

#### FR-19 — Резервирование позиции [M]

**Описание:** Зритель может зарезервировать позицию (занять как подарок).

**Поведение:**
- Нажать «Зарезервировать» → если не авторизован и нет гостевой сессии → сначала FR-21 (ввод имени)
- `POST /items/{itemId}/reserve`
- Оптимистичное обновление: кнопка меняется на «Отменить», статус «Зарезервировано»
- При ошибке (уже занято) → откат + toast «Кто-то успел раньше»
- При успехе → toast «Зарезервировано!» + haptic feedback

---

#### FR-20 — Снятие резервации [M]

**Описание:** Зритель может отменить свою резервацию.

**Поведение:**
- Нажать «Отменить резервацию» → Alert «Снять резервацию?»
- `DELETE /items/{itemId}/reserve`
- Оптимистичное обновление: статус «Свободно»

---

#### FR-22 — Участие в групповой покупке [M]

**Описание:** Зритель может внести вклад в групповую покупку.

**BottomSheet «Участвовать»:**
- Поле «Сумма»: числовое, > 0
- Поле «Сообщение»: необязательное, до 200 символов
- Кнопка «Внести вклад»

**Поведение:**
- `POST /items/{itemId}/contributions { amount, note }`
- Прогресс-бар обновляется оптимистично
- Toast «Вклад внесён! 🎉»
- Пользователь видит свой вклад в истории (GetSummary)

---

#### FR-23 — Удаление своего вклада [S]

**Описание:** Участник может удалить свой вклад в групповую покупку.

**Поведение:**
- В BottomSheet с деталями — список вкладов пользователя
- Свайп или кнопка «Удалить» → Alert → `DELETE /items/{itemId}/contributions/{id}`

---

### 5.6. Гостевая сессия

#### FR-21 — Создание гостевой сессии [M]

**Описание:** Незарегистрированный пользователь может взаимодействовать со списком, представившись именем.

**Поведение:**
- При попытке зарезервировать или внести вклад без авторизации → показать BottomSheet
- BottomSheet «Как вас зовут?»:
  - Поле «Имя»: обязательное, 2–50 символов
  - Кнопка «Продолжить»
  - Ссылка «Войти в аккаунт»
- При нажатии «Продолжить» → `POST /auth/guest { display_name }` → сохранить guest_token в MMKV → продолжить действие
- GuestBanner остаётся видимым в верхней части экрана: «Вы входите как [Имя] · Войти»

---

#### FR-24 — Persistent гостевая сессия [M]

**Описание:** Гостевая сессия сохраняется между запусками приложения.

**Поведение:**
- guest_token хранится в MMKV
- При следующем открытии публичного списка — токен автоматически отправляется
- GuestBanner показывается с сохранённым именем
- Кнопка «Войти» в GuestBanner → LoginScreen

### 5.7. Профиль пользователя

#### FR-25 — Просмотр профиля [M]

**Описание:** Пользователь видит свой профиль.

**Содержимое экрана:**
- Аватар (FastImage, с fallback на инициалы)
- Отображаемое имя
- Email (не редактируется)
- Кнопка «Выйти»

---

#### FR-26 — Редактирование имени [S]

**Описание:** Пользователь может изменить отображаемое имя.

**Поведение:**
- Tap на имя → inline edit или отдельный AlertPrompt
- `PATCH /auth/me { display_name }` → обновить имя в UI

---

### 5.8. Поделиться списком

#### FR-27 — Шеринг публичной ссылки [M]

**Описание:** Владелец может поделиться ссылкой на свой список.

**Поведение:**
- Нажать иконку «Поделиться» на ListDetailScreen
- `Share.share({ message: 'Мой список желаний: https://wishify.app/l/{slug}' })`
- Открывает нативный iOS Share Sheet
- Ссылка копируется/отправляется выбранным способом

---

### 5.9. Real-time обновления

#### FR-28 — WebSocket подключение [M]

**Описание:** При просмотре списка (публичного или owner view) — изменения видны без перезагрузки.

**Поведение:**
- При открытии ListDetailScreen или PublicListScreen → подключить WebSocket
- При закрытии экрана → отключить WebSocket
- При уходе в фон (AppState = background) → WebSocket может быть разорван iOS
- При возврате в foreground → переподключиться
- При потере соединения → автоматический reconnect (exponential backoff: 1с → 2с → 4с → ... → 30с)
- Обрабатываемые события: `item.created`, `item.updated`, `item.deleted`, `item.reordered`, `reservation.changed`, `contribution.added`, `contribution.removed`, `list.updated`
- При получении события → инвалидировать соответствующий Query cache → React Query перефетчит

## 13. Ограничения и допущения

### 13.1. Технические ограничения

1. **Refresh token при cold start:** Бэкенд хранит refresh token в httpOnly cookie. На мобильном устройстве cookie jar in-memory (не персистируется). Если access token истёк после перезапуска приложения и бэкенд не поддерживает `X-Client: mobile` — пользователь будет разлогинен. Это известный trade-off.

2. **iOS background WebSocket:** iOS разрывает WebSocket-соединения при уходе в фон. Приложение переподключается при возврате в foreground, но события, произошедшие в фоне, будут получены только после переподключения (Query invalidation на reconnect).

3. **Google OAuth redirect URI:** Нужно добавить `wishify://oauth/callback` в Google Cloud Console. Это изменение нужно сделать до тестирования OAuth.

4. **Android вне scope:** Все нативные зависимости выбраны с поддержкой Android, но сборка, тестирование и деплой под Android — вне данного ТЗ.

### 13.2. Допущения

- Бэкенд стабилен и соответствует описанию API в `MOBILE_APP_GUIDE.md`
- Railway и Vercel доступны (SLA ≥ 99%)
- Apple Developer Program аккаунт активен
- Mac с Xcode 15+ доступен для разработки и сборки
